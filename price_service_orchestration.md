# 가격 서비스 오케스트레이션 설계

## 1. 목표

이 문서는 분산된 가격 조회 로직을 `utils/price_service.py`로 통합하기 위한 구현 기준을 정의한다.

목표는 아래 3가지다.

1. 실시간 가격과 환율 조회의 단일 진입점을 만든다.
2. 시장 상태에 따라 TTL을 다르게 적용해 불필요한 외부 호출을 줄인다.
3. 화면 표시용 실시간값과 전략 계산용 종가 데이터를 분리해 정합성을 유지한다.

## 2. 현재 문제

현재 가격 조회 로직은 여러 파일에 분산되어 있다.

- `app.py`
  - 환율을 직접 조회하고 Streamlit 캐시를 별도로 사용한다.
- `utils/rankings.py`
  - 국가별 실시간 스냅샷을 직접 조회한다.
- `utils/portfolio_io.py`
  - 한국 보유 종목 실시간 오버레이와 환율 조회를 직접 처리한다.
- `app_pages/etf_market_page.py`
  - 네이버 ETF 스냅샷을 직접 조회한다.

이 구조의 문제는 아래와 같다.

1. 동일한 시점에 같은 API를 여러 번 호출한다.
2. TTL 정책이 호출부마다 달라진다.
3. 실패 처리 방식이 제각각이라 디버깅이 어렵다.
4. 실시간값이 전략 계산 로직으로 과도하게 스며들 위험이 있다.

## 3. 설계 원칙

이 작업에서는 아래 원칙을 반드시 지킨다.

1. 가격 조회는 `utils/price_service.py`만 외부 API를 직접 호출한다.
2. `utils/data_loader.py`는 원천 fetch 함수만 유지하고, 오케스트레이션은 담당하지 않는다.
3. 실시간값은 표시 보강 용도로만 사용하고, 순위 계산의 기준 데이터는 기존 캐시 종가 시계열이다.
4. 필수 입력값이 없으면 fallback 없이 즉시 명확한 에러를 발생시킨다.
5. 조회 실패 시 조용히 다른 소스로 우회하지 않는다.
6. stale 데이터 재사용 여부는 서비스에서 명시적으로 판단하고 호출부에 드러나야 한다.
7. 최신 거래일 판단 기준은 기존 문서 원칙대로 한국 날짜 기준을 따른다.

## 4. 서비스 책임 범위

`utils/price_service.py`는 아래 책임만 가진다.

### 4.1 실시간 가격 조회

- 한국 ETF 실시간 스냅샷
- 한국 개별 종목 실시간 스냅샷
- 호주 ETF 실시간 스냅샷

### 4.2 환율 조회

- USD/KRW
- AUD/KRW

### 4.3 캐시 및 TTL 관리

- 프로세스 메모리 캐시
- 시장 상태별 TTL 계산
- stale 허용 여부 판단
- 호출 결과 메타데이터 기록

## 5. 비범위

이번 작업에서 아래는 `price_service.py`의 직접 책임으로 두지 않는다.

1. Parquet 가격 캐시 저장
2. 장기 OHLCV 수집
3. 종목 메타데이터 관리
4. Streamlit UI 포맷팅

단, 기존 코드에서 실시간 현재가로 마지막 행을 임시 보정하는 로직은 서비스 도입 후에도 호출부에 남길 수 있다. 이 경우에도 실시간 원천 데이터는 서비스에서만 가져온다.

## 6. 공개 인터페이스 초안

서비스 외부 공개 함수는 최소한 아래 수준으로 시작한다.

```python
get_realtime_snapshot(country_code: str, tickers: Sequence[str]) -> dict[str, dict[str, float]]
get_exchange_rates() -> dict[str, Any]
clear_price_service_cache() -> None
```

필요하면 아래 보조 함수를 추가한다.

```python
get_market_status(country_code: str) -> MarketStatus
get_realtime_cache_meta(cache_key: str) -> dict[str, Any] | None
```

초기 버전에서는 공개 인터페이스를 과도하게 늘리지 않는다.

## 7. 캐시 구조

캐시는 국가 단위 뭉텅이 캐시보다 요청 단위를 반영하는 구조가 안전하다.

권장 구조:

```python
_PRICE_SERVICE_CACHE = {
    "realtime:kor:069500,357870": {
        "data": {...},
        "fetched_at": datetime,
        "expires_at": datetime,
        "is_stale": bool,
        "source": "naver_etf|naver_stock|mixed",
    },
    "realtime:au:acdc,mnrs": {
        ...
    },
    "fx:usd_aud": {
        ...
    },
}
```

핵심 기준:

1. 티커 집합은 정규화 후 정렬해서 캐시 키를 만든다.
2. 반환 직전에는 요청한 티커만 필터링한다.
3. 캐시 엔트리에는 `data` 외에 `fetched_at`, `expires_at`, `is_stale`, `source`를 함께 저장한다.
4. stale 재사용이 발생하면 로그에서 구분 가능해야 한다.

## 8. TTL 정책

TTL은 시장 상태에 따라 계산한다.

### 8.1 한국 실시간 가격

- 장중: 30초
- 장 시작 직전 완충 구간: 30초
- 장 종료 직후 완충 구간: 30초 또는 60초
- 장 마감 후: 1시간
- 비거래일: 1시간

### 8.2 호주 실시간 가격

- 장중: 60초
- 장 시작 직전 완충 구간: 60초
- 장 종료 직후 완충 구간: 60초
- 장 마감 후: 1시간
- 비거래일: 1시간

### 8.3 환율

- 기본 TTL: 1시간
- 단, 장중 짧은 TTL로 내리더라도 호출부별 캐시를 따로 두지 않고 서비스 캐시만 사용한다.

## 9. 실패 처리 원칙

실패 처리는 애매하게 두지 않는다.

1. 실시간 조회 실패 시 빈 dict를 조용히 만드는 로직은 최소화한다.
2. 이전 성공 캐시가 있고 stale 재사용이 허용되는 경우에만 stale 데이터를 반환한다.
3. stale 데이터를 반환했다면 `is_stale=True`가 남아야 한다.
4. 이전 캐시가 없고 조회도 실패하면 호출부가 처리할 수 있도록 명확한 실패 상태를 반환하거나 예외를 발생시킨다.

초기 구현에서는 아래 방식이 현실적이다.

- 순위/포트폴리오/ETF 마켓용 실시간 스냅샷:
  - 실패 시 stale 캐시가 있으면 반환
  - 없으면 빈 결과 반환
- 환율:
  - stale 캐시가 있으면 반환
  - 없으면 예외 또는 명확한 0.0 금지 정책 중 하나를 선택

환율의 경우 `0.0`을 정상값처럼 내려주는 현재 패턴은 제거 대상이다.

## 10. 호출부 이관 대상

아래 파일은 직접 API 호출을 제거하고 서비스만 사용하도록 바꾼다.

### 10.1 `app.py`

- 환율 조회 로직 제거
- `st.cache_data` 기반 환율 캐시 제거
- `price_service.get_exchange_rates()` 사용

### 10.2 `utils/rankings.py`

- `_load_realtime_snapshot()` 내부 직접 fetch 제거
- `price_service.get_realtime_snapshot()` 사용

### 10.3 `utils/portfolio_io.py`

- 한국 보유 종목 실시간 오버레이 조회를 서비스로 이관
- 환율 조회를 서비스로 이관
- Streamlit 환율 캐시 제거

### 10.4 `app_pages/etf_market_page.py`

- 네이버 ETF 스냅샷 직접 호출 제거
- `price_service.get_realtime_snapshot("kor", tickers)` 사용

## 11. 순위 계산 정합성 규칙

이 부분은 문서로 못 박아야 한다.

1. 순위 계산의 기준 시계열은 캐시된 종가 데이터다.
2. 실시간값은 `현재가`, `일간(%)`, `NAV`, `괴리율` 같은 표시 컬럼 보강에 사용한다.
3. 실시간값으로 마지막 행을 임시 생성하거나 덮어쓸 수는 있지만, 이것이 장기 캐시에 저장되면 안 된다.
4. 실시간값이 없더라도 순위 계산은 기존 캐시 종가 기준으로 계속 가능해야 한다.

## 12. 구현 순서

권장 순서는 아래와 같다.

1. `utils/price_service.py` 신규 구현
2. `utils/rankings.py` 이관
3. `utils/portfolio_io.py` 이관
4. `app_pages/etf_market_page.py` 이관
5. `app.py` 환율 표시 이관
6. 중복 캐시 제거 및 정리

이 순서를 권장하는 이유는 `rankings`와 `portfolio_io`가 가장 큰 중복 호출 지점이기 때문이다.

## 13. 검증 항목

수정 후 최소한 아래를 검증한다.

1. 동일 요청 반복 시 TTL 안에서는 외부 API 재호출이 줄어드는지 확인
2. 장 마감 후 동일 페이지 재진입 시 실시간 API 호출이 과도하게 반복되지 않는지 확인
3. 순위 페이지에서 `현재가`와 `일간(%)`는 실시간 반영되고 점수 계산은 깨지지 않는지 확인
4. 보유 종목 화면에서 환율과 현재가 계산이 유지되는지 확인
5. ETF 마켓 페이지가 기존처럼 실시간 컬럼을 채우는지 확인
6. stale 데이터 사용 시 로그와 메타데이터가 남는지 확인

## 14. 오픈 이슈

구현 전에 아래는 최종 결정이 필요하다.

1. 환율 조회 실패 시 예외를 올릴지, stale만 허용할지
2. 캐시 키를 티커 집합 단위로 둘지, 티커별 개별 엔트리로 둘지
3. 실시간값 메타데이터를 호출부 UI에 노출할지, 내부 로그로만 남길지

현재 기준에서는 아래 선택을 권장한다.

- 환율: stale 우선, stale도 없으면 예외
- 캐시 키: 티커 집합 단위로 시작
- 메타데이터 노출: 우선 내부 로그만

## 15. 결론

이번 작업의 핵심은 단순히 캐시를 추가하는 것이 아니다.

가격 조회 책임을 한 곳으로 모으고, 실시간값의 역할을 표시 보강으로 제한하고, TTL 정책을 일관되게 적용하는 것이 핵심이다.

구현은 `utils/price_service.py` 중심으로 진행하되, 호출부의 직접 조회와 Streamlit 캐시는 함께 제거한다.
