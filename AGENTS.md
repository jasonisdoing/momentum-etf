# AGENTS.md

이 문서는 이 저장소에서 작업할 때 항상 우선 참고하는 로컬 작업 규칙이다.
애매할 때는 "짧고 실행 가능하며 검증 가능한 규칙"으로 해석한다.

## 1. 최우선 규칙

1. 모든 응답은 한국어로 작성한다.
2. 내가 요구하는 걸 완벽하게 이해할 때까지 질문해. 그리고 100% 이해했을 때 코드짜기를 시작한다.
3. 코드 주석은 한국어로 작성한다.
4. 사용자가 "질문"을 한 경우에는 바로 수정하지 말고 먼저 답변한다.
5. 질문에 답하는 과정에서 수정 필요 사항을 발견하면, 답변 후 문제점과 수정 필요 여부를 함께 알린다.
6. 사용자가 명시적으로 수정이나 구현을 요청한 경우에는 별도 계획 문서보다 실제 코드 수정과 검증을 우선한다.

## 2. 작업 전 확인

1. 작업이 새로운 모듈/도메인이거나 사용자가 README/문서를 명시적으로 참조한 경우에만
   `README.md` 및 관련 문서를 읽는다. 이미 알고 있는 영역의 단순 수정에는 생략한다.
2. `znotes/` 폴더의 파일은 사용자 메모로 간주하고, 어떤 경우에도 읽어나 참조하지 않는다.
3. 리팩토링이나 구조 변경 시 `Dockerfile`, 배포 스크립트, 실행 엔트리 포인트 영향 여부를 반드시 확인한다.

## 3. 구현 원칙

1. 암묵적인 fallback, silent default, 임의 보정 로직을 만들지 않는다.
2. 함수 시그니처에 의미 있는 기본값을 넣어 동작을 암묵적으로 결정하지 않는다.
3. 필수 설정값이 없으면 즉시 명확한 에러를 발생시킨다.
4. 함수는 가능하면 짧고 한 가지 역할만 하게 유지한다.
5. 하나의 함수가 여러 책임을 가지면 하위 함수로 분리하는 쪽을 우선 검토한다.
6. 문법 호환성 문제를 이유로 코드 수준을 낮추지 않는다.
7. 예를 들어 실행 환경 문제를 맞추기 위해 최신 타입 힌트를 구문상 구버전 스타일로 되돌리는 식의 수정은 하지 않는다.
8. 대신 올바른 실행 환경과 프로젝트가 의도한 Python 버전을 찾는다.

## 4. 실행 및 검증

1. Python 실행 시 시스템 `python`, `python3`를 당연하게 가정하지 않는다.
2. 먼저 프로젝트의 가상환경 경로를 확인하고 그 Python을 우선 사용한다.
3. 코드 수정이 ① 다중 파일 변경 ② 타입/시그니처 변경 ③ 새 의존성 추가 중 하나에 해당할 때만
   `.pre-commit-config.yaml` 기반 검증을 실행한다. 한두 줄 수정이나 CSS/문구 수정은 생략한다.
4. 검증 명령은 가능한 한 수정 범위와 직접 관련된 것부터 최소 단위로 실행한다.

## 5. 문서 동기화

1. 사용자가 문서 갱신을 명시적으로 요청하거나, UI/CLI 사용법·아키텍처가 실제로 바뀐 경우에만
   관련 문서를 갱신한다. 그 외에는 문서를 읽지 않는다.

## 6. 작업 방식

1. 기본 원칙은 순차 처리다.
2. Frontend와 Backend 개발이 동시에 있을때는 UI를 먼저 개발하고 확인을 받은 후 Backend 개발을 진행한다.
3. Implementation Plan, Task, Walkthrough 같은 중간 산출물 문서는 기본적으로 만들지 않는다.
4. 다만 사용자 요청 또는 작업상 꼭 필요한 경우에만 짧게 작성한다.

## 7. Git 규칙

1. 사용자가 명시적으로 요청하지 않으면 `git commit`과 `git push`는 실행하지 않는다.
2. 코드 변경 작업을 마치면 바로 사용할 수 있는 짧은 커밋 메시지 스타일 요약을 함께 제공한다.
3. 커밋 요약은 한국어로, 가능하면 20단어 이내로 작성한다.

## 8. 해석 우선순위

1. 사용자의 현재 요청
2. 이 `AGENTS.md`
3. `README.md`와 거기서 언급된 관련 문서
4. 그 외 코드와 기존 구현 관례

## 9. 토큰 효율 (플랜 사용량 절감)

1. 같은 대화 안에서 이미 읽은 파일은 다시 읽지 않는다. 변경 후에도 Edit/Write가 성공했다면
   재확인 Read를 생략한다.
2. 탐색은 좁게 시작한다. 디렉터리 일괄 Read 대신 `rg`/Grep으로 키워드부터 좁히고,
   필요한 부분만 offset/limit로 부분 Read 한다.
3. Agent(서브에이전트) 호출은 사용자가 명시 요청하거나, 대화 컨텍스트로는 도저히 답할 수 없는
   광범위 탐색일 때만 사용한다. 단순 파일 찾기·수정은 직접 처리한다.
4. WebSearch/WebFetch는 사용자가 외부 자료를 명시 요청한 경우에만 호출한다.
5. 코드 수정 후의 "검증용" 빌드/테스트 전체 실행은 사용자가 요청하거나 회귀 위험이 있을 때만
   수행한다. 단순 UI 텍스트·CSS·상수 변경은 생략한다.
6. 응답에서 변경 내용을 다시 코드블록으로 통째로 복사하지 않는다. 파일 경로와 핵심 diff 요약만 남긴다.
7. 동일 작업 내 도구 호출은 가능하면 한 메시지에서 병렬로 묶어 호출한다 (왕복 토큰 절감).
8. 모호하면 사용자에게 한 줄로 짧게 확인 질문을 한다 — 추측 후 잘못 구현해 다시 작업하는 비용보다 저렴하다.

## 10. 이 문서의 의도

1. 쓸데없이 긴 설명보다 바로 수정 가능한 판단 기준을 제공한다.
2. 모호한 규칙보다 실제 작업 중 충돌이 적은 규칙을 남긴다.
3. 새 규칙을 추가할 때는 "언제", "무엇을", "예외는 무엇인지"가 바로 드러나게 쓴다.
