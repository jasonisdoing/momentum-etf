# 05. 다음 작업 순서

## 바로 다음 액션

로컬 하이브리드 구현과 1차 메뉴 이전은 끝났다.
이제 다음 작업은 `배포 테스트`다.

## Step 1. 배포 파일 전략 확정

목표:

- 기존 운영용 `docker-compose.yml` 를 유지할지, 하이브리드용 새 배포 파일로 분리할지 결정한다.
- 운영 도메인 `etf.dojason.com` 아래에서 `/` 는 Node, `/py/*` 는 Python으로 분기할 수 있는 운영용 프록시 구성을 만든다.

검증 포인트:

- 운영 서버의 `nginx-proxy` 구조와 충돌하지 않는가
- 배포 파일만으로 Node/Python/프록시 세 컨테이너가 함께 올라갈 수 있는가

## Step 2. 스테이징 또는 안전한 운영 테스트

목표:

- 실제 운영 서버에서 하이브리드 구성을 한 번 띄운다.
- 가능하면 운영 교체 전 별도 테스트 주소나 안전한 방식으로 먼저 확인한다.
- 현재 저장소 기준으로는 `upgrade` 브랜치 push -> GitHub Actions Oracle VM 배포 경로를 사용한다.

검증 포인트:

- `/`
- `/dashboard`
- `/cash`
- `/snapshots`
- `/market`
- `/py/`
- `/py/rank`
- 컨테이너 재기동 후에도 502 없이 다시 열리는가

## Step 3. 배포 후 회귀 확인

목표:

- 배포 후 Node 1차 메뉴와 Python 핵심 화면이 모두 유지되는지 본다.

검증 포인트:

- Node 화면에서 실제 데이터가 계속 정상 표시되는가
- Python 쪽 분석/순위 메뉴가 깨지지 않는가
- 자산관리 양방향 저장이 유지되는가

## Step 4. 그 다음 2차 개선

목표:

후보:

- 대시보드 카드/표 구성 추가 정리
- 다음 Node 이관 대상 선정
