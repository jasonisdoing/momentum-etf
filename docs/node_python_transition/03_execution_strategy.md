# 03. 실행 전략

## 왜 문서보다 기술 검증이 먼저인가

이번 작업은 최종 설계보다 먼저 다음 리스크를 확인해야 한다.

- nginx가 단일 Node 진입 구조에서 안정적으로 동작하는가
- 초기 공존 검증이 실제로 가능한가
- Node 앱이 같은 도메인에서 공존할 때 정적 자원/라우팅 문제가 없는가
- 현재 Docker 구조를 크게 깨지 않고 두 앱을 띄울 수 있는가

이 검증이 되지 않으면 이후 계획은 쉽게 되돌려질 수 있다.

## 1차 전략

### 전략 이름

`최소 공존 버전 스파이크`

### 구성

- 새 Node.js + TypeScript 앱을 최소 화면부터 추가
- 초기에는 Python 공존도 검증했지만, 최종 목표는 Node 단일 UI다

### 이번에 검증할 것

- Node 앱이 뜬다
- 초기 공존 검증이 가능하다
- 이후 Node 단일 UI로 정리할 수 있다
- 최소 라우팅이 실제로 된다

## 1차 검증 범위

### Node 쪽

- 더미 또는 최소 화면
- `/dashboard`
- `/cash`
- `/snapshots`
- `/market`

## 이번 단계에서 요구되는 산출물

- Node 앱 최소 실행 구조
- 초기 공존 검증을 위한 실행 방법
- nginx 경로 분기 설정
- 로컬 또는 운영과 유사한 환경에서의 실제 실행 확인

## 이번 단계에서 아직 미루는 것

- 실제 API 설계 세부화
- Node 화면 완성도 향상
- 메뉴 이전 순서의 세부 일정화
- 최종 인증 고도화 이후의 운영 정책

## Node UI 구현 원칙

Node 쪽 화면은 앞으로 `ERP 스타일 기준`으로 구현한다.

- 업무 화면처럼 바로 쓰는 느낌을 우선한다.
- 큰 히어로 섹션, 과한 소개 문구, 랜딩 페이지식 카드 나열은 지양한다.
- 표, 필터, 압축형 입력 폼, 고정 메뉴 구조를 선호한다.
- 특별한 요청이 없으면 카드보다 테이블을 기본 선택으로 사용한다.
- 한 화면에서 조회와 수정이 빠르게 이어지게 구성한다.
- 구현 기준 라이브러리는 `Tabler`로 통일한다.
- Navbar, page header, card, table, button, form control은 가능하면 `Tabler 기본 컴포넌트와 클래스`를 우선 사용한다.
- 커스텀 CSS는 꼭 필요한 도메인 의미색과 최소한의 보조 규칙만 남긴다.
- 화면마다 보이는 높이/간격 차이를 페이지별 수동 보정으로 해결하지 않고, `전역 Tabler 기준선`을 먼저 고친다.

## 현재 실행 원칙

- 로컬 개발은 기본적으로 Docker 없이 `npm run dev` 중심으로 진행한다.
- Docker 하이브리드는 프록시/배포 구조 회귀가 필요할 때만 다시 사용한다.
- Python과 Node를 서로 이어주는 편의 기능보다, Python 기능을 Node로 더 빨리 이관하는 쪽을 우선한다.
- Node 인증은 `Google OAuth + 허용 이메일 화이트리스트`를 사용한다.
