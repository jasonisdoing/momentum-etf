# 03. 실행 전략

## 왜 문서보다 기술 검증이 먼저인가

이번 작업은 최종 설계보다 먼저 다음 리스크를 확인해야 한다.

- nginx가 `/`와 `/py`를 다른 앱으로 안정적으로 분기할 수 있는가
- Streamlit이 `/py/*` 경로 구조에서 정상 동작하는가
- Node 앱이 같은 도메인에서 공존할 때 정적 자원/라우팅 문제가 없는가
- 현재 Docker 구조를 크게 깨지 않고 두 앱을 띄울 수 있는가

이 검증이 되지 않으면 이후 계획은 쉽게 되돌려질 수 있다.

## 1차 전략

### 전략 이름

`최소 공존 버전 스파이크`

### 구성

- 기존 Python(Streamlit) 앱은 유지
- 새 Node.js + TypeScript 앱은 최소 화면만 추가
- nginx가 경로를 분기

### 이번에 검증할 것

- Node 앱이 뜬다
- Python 앱이 `/py/*` 아래에서 뜬다
- 둘이 같은 도메인에서 공존한다
- 최소 라우팅이 실제로 된다

## 1차 검증 범위

### Node 쪽

- 더미 또는 최소 화면
- `/dashboard`
- `/cash`
- `/snapshots`
- `/market`

### Python 쪽

- 기존 앱
- 최소 1~2개 메뉴만 `/py/*` 구조로 정상 확인
- 예: `/py/rank`, `/py/system`

## 이번 단계에서 요구되는 산출물

- Node 앱 최소 실행 구조
- 두 앱을 함께 띄우는 실행 방법
- nginx 경로 분기 설정
- 로컬 또는 운영과 유사한 환경에서의 실제 실행 확인

## 이번 단계에서 아직 미루는 것

- 실제 API 설계 세부화
- Node 화면 완성도 향상
- 메뉴 이전 순서의 세부 일정화
- 인증 공유 구조 최적화
